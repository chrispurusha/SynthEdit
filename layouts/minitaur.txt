# minitaur.txt — Moog Minitaur device descriptor.
#
# manufacturerId: 0x04 (Moog) — shared with voyager.txt; matches byte 2 of a
# real Minitaur patch-dump SysEx captured on real hardware in an earlier
# session (see voyager.txt's own manufacturerId comment).
#
# modelId: there's no separate config field for this (it's Moog's own
# proprietary third header byte, not part of the Universal Identity Reply
# scheme manufacturerId/familyId/memberId model), so it only ever appears
# inline in stateRequestSysEx below — 0x08. CONFIRMED, and this is exactly
# what distinguishes this file from voyager.txt's byte at the same position
# (0x01): sending 0x08 gets a genuine Minitaur reply every time; sending
# voyager.txt's 0x01 with the same 0x1B function code gets nothing back from
# the Voyager (tested 2026-07-06 — see voyager.txt's stateRequestSysEx
# comment for the full story of how these two got disambiguated after an
# initial mixup with both devices live at once).
#
# familyId / memberId: no informed guess — left as 0x00 placeholders, moot
# in practice since identityQuery is no (see below).
#
# identityQuery no: Moog gear doesn't answer a Universal Device Inquiry at
# all — same finding as voyager.txt, same manufacturer, presumably a shared
# trait across Moog's non-identity-answering analog line. Connects using
# midiChannel/midiPort below directly (see connect_without_identity() in
# midiComms.c).
#
# midiChannel 11: observed directly from this Minitaur's own traffic — every
# stateRequestSysEx reply captured (two separate sessions, 2026-07-06) came
# back with the same channel nibble (status byte 0xBA = Control Change,
# channel 11 1-indexed). Not verified against the unit's own front-panel MIDI
# setting the way voyager.txt's channel 8 was (by turning a knob and watching
# the resulting live CC) — this one comes only from the dump reply's own
# channel byte. Matters less than usual anyway: dispatch_cc() in midiComms.c
# auto-corrects gDevice.id to whatever channel a no-identity device's own
# traffic actually uses the first time anything arrives, so this is only ever
# a first guess for sending before we've heard anything back.
#
# midiPort: this Minitaur enumerates as its own named CoreMIDI port ("Moog
# Minitaur") rather than sitting behind a separate MIDI interface the way the
# Voyager sits behind "Elektron TM-1" — matched case-insensitively as a
# substring against each destination's display name (see
# find_destination_by_name() in midiComms.c). Adjust if the unit is ever
# renamed or moved to a different interface.
#
# stateRequestSysEx F0 04 08 1B 00 00 F7 — documented on Moog's own forum
# (thread title: "Minitaur feature request") as a command that makes the
# Minitaur dump its current CC value for every internal parameter.
# CONFIRMED genuinely from this Minitaur specifically: resent with a
# source-tagged listener (distinguishing which physical MIDI port a reply
# actually came from, after the mixup mentioned above) and every single reply
# was tagged "from Moog Minitaur" — none from anywhere else. Sent once via
# midi_send() right after connect_without_identity() connects (see
# midiComms.c); replies arrive as ordinary CC messages through the normal
# dispatch_cc() path, same as any live knob move — so the panel below
# populates from real hardware state on connect rather than showing defaults
# until every control is physically touched.
#
# Dial map: same CC numbers as voyager.txt (Ctrlr-panel-derived — see that
# file's own comment for the full provenance/parsing story), which makes
# sense given both instruments share the same underlying Moog analog synth
# engine. Cross-validated harder here than on the Voyager side, though:
# of these 51 dials, 34 (22 of 28 fourteen-bit pairs, 12 of 23 single-byte
# ones) came back with live values in the captured dump reply above — direct
# confirmation those CCs genuinely exist on this hardware, not just inferred
# from a third-party panel. The other 17 didn't appear in that particular
# dump (missing: mwAmount, pedalAmount, osc1Wave, osc2Wave, osc2Freq,
# extLevel, lfoSync, releaseSwitch, mwSource, mwDestination, osc1Octave,
# osc2Octave, osc3Octave, oscSync, oscFM, osc3KB, osc3Toggle) — could be
# state-dependent (only touched/non-default params get echoed), or genuinely
# not present on a Minitaur (no keybed, so a true "mod wheel" may not apply
# the same way it does on Voyager, for instance). Labels below are inherited
# from the Voyager-oriented Ctrlr panel too, so treat the wording (not the CC
# numbers) as unconfirmed against this unit's actual front-panel labelling.
