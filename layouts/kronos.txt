# kronos.txt — Korg Kronos device descriptor. First pass only: identity +
# live "Sync from synth" + AL-1 Filter A/B Cutoff, both EXi slots, all via
# SysEx Parameter Change — owner's own explicit scope ("doesn't have to be
# everything... perhaps just filters", 2026-07-14). The Kronos's own
# architecture (Combi/Program/Sequencer modes, multiple Timbres/Tracks per
# Combi, banked Programs/Combinations/Set Lists, EXi/HD-1/AL-1/CX-3/...
# synthesis engines) is vastly larger than Z1's — this file deliberately
# does NOT attempt a Program Dump protocol the way z1.txt does, but DOES
# use the same generic Parameter Change addressing (typ=/soc=/sub=/pid=/
# idx=, this device's own SysEx analogue of z1.txt's group=/param=).
#
# NO CC DIALS — deliberate, not just untried. An earlier version of this
# file had a "Filters" page using the front-panel "Realtime Knobs" CCs
# (CC 74/71/79/72, from the MIDI Implementation doc's own table). Owner's
# call 2026-07-14: "I don't think we should be dealing with cc on the
# korgs, for the same reason we have problems on the Z1. Sysex for editing
# is much more reliable." — z1.txt itself already follows this (grep it:
# not one dial uses cc=, every single one is group=/param=), so this file
# now matches that same house rule instead of being the one exception.
# Removed rather than converted 1:1: the Realtime Knobs are an
# engine-agnostic modulation-routing layer (same CC regardless of which of
# the 8 EXi engines/HD-1 is loaded), so there's no single equivalent
# SysEx address the way there is for a specific engine's own parameter
# (below) — building that out per-engine is a real scope increase, not
# something to default into. If real-time-knob-style generic control is
# wanted later, that's a separate ask, not a fallback default.
#
# Identity: standard Universal Identity Reply, confirmed ANSWERED (unlike
# Voyager/Minitaur, which never reply at all) — see "2-4-1 ... DEVICE
# INQUlRY (When received this message, transmit...)" in the MIDI
# Implementation doc's own SYSTEM EXCLUSIVE section. memberId depends on
# keybed size: 61-key=0x05, 73-key=0x0E, 88-key=0x17 (per the doc's own
# DEVICE INQUIRY REPLY byte table) — 0x05 below is the 61-key, per the
# owner's own hardware.
device          Korg Kronos
description     First pass: live "Sync from synth" + AL-1 Filter A/B Cutoff (both EXi slots, SysEx Parameter Change) — no CC dials, by design (see header)
manufacturerId  0x42
familyId        0x68
memberId        0x05
scrollDial      al1FilterCutoff
progNameLen     24

# Kronos does NOT speak Z1's own Program Data Dump Request/Program Write
# Request protocol (func 0x1C/0x11) — it has its own, completely different
# object-based SysEx protocol (Object Dump / Current Object Dump / Set
# Current Object / Parameter Change, obj+bank+TYP/SOC/SUB/PID/IDX addressing
# — see KRONOS_MIDI_SysEx.txt, Korg's official "SysEx Documentation 2.1",
# a SEPARATE download from the compact MIDI Implementation chart). Without
# this, the app's generic Korg-style background prefetch/Load-Store-picker/
# Backup-Restore-Bank-(Individual-Files) machinery — all built around Z1's
# specific protocol — would fire at connect and retry forever with zero
# replies, exactly as found 2026-07-14 connecting a real Kronos for the
# first time. See supportsKorgProgramDump's own comment, panelConfig.h.
supportsKorgProgramDump no

# ── AL-1 filter section ──────────────────────────────────────────────────────
# Parameter Change (SysEx), not CC — CONFIRMED WORKING against real hardware
# 2026-07-14 (owner: "Cutoff is working for the first synth slot and The
# Sublime Mariner patch"; both slots and both filters subsequently confirmed
# too) — turning these dials audibly changes the live sound, AND the app now
# decodes the Kronos's own outgoing Parameter Change (func 0x43) when a
# physical front-panel/object-editor control is turned, so these dials track
# real hardware live in both directions (handle_kronos_parameter_change(),
# synthComms.c). Only means anything while an AL-1-engine EXi Program is
# actually loaded in the targeted slot; meaningless (inert, or hitting an
# unrelated parameter) for any other engine's programs.
#
# Filter A targets "[Filter A] Cutoff" in AL-1's own "Analog 4-Pole Filter"
# component: SOC=20 (component number — see the "Analog 4-Pole Filter
# (comp 20, sub 0)" table, SysExParams/VoiceModels/AL-1.txt), SUB=0, PID=4
# (the component's own param index for Cutoff), IDX=0, range 0..99 per that
# same table. TYP selects which of the Kronos's two layerable EXi engine
# slots a Program can use — 11 for slot 1, 12 for slot 2 (confirmed by the
# consistent "11/12" pairing throughout Prog_EXi.txt's own TYP column,
# e.g. every "[LFO1]"/"[Filter]"/etc. entry lists both together) — a
# Program can layer two independent EXi instances (see e.g. "J.Lorber
# MarkV EP w/Pad" from the earlier U-AA sampling: EXi1=EP-1, EXi2=MOD-7),
# so both need their own dial to reach either one. Both EXi1 and EXi2
# confirmed independently on real hardware 2026-07-14.
#
# Filter B Cutoff (PID=49, same SOC/SUB/IDX pattern as Filter A) was
# discovered by PASSIVE CAPTURE, not the doc: AL-1 has two independent
# filters (A and B); turning the real front-panel knobs on each in turn
# (owner's own order: "exi1 filter A, then exi1 filter B, then exi2 filter
# A, then exi2 filter B") showed the Kronos itself echoing live Parameter
# Change SysEx (func 0x43) — captured with tools/kronos_monitor (passive
# MIDI listener, no send). PID=4 matched our existing Filter A wiring
# exactly, confirming the addressing scheme is right; PID=49 was new and
# is Filter B, not Resonance as first guessed from the (since-lost) doc
# reference — trust the hardware capture over an unconfirmed doc reading,
# same principle as the self-check-vs-hardware lesson from the Filter A
# work. Not yet cross-checked against the doc (the downloaded "SysEx
# Documentation 2.1" reference is no longer on disk this session).
#
# Source for Filter A: Prog_EXi.txt (KORG's official "SysEx Documentation
# 2.1" download, KRONOS_MIDI_SysEx.txt's own companion files — a separate,
# much larger reference from the compact MIDI Implementation chart at the
# top of this file). Source for Filter B: live hardware capture only, see
# above.
#
# dumpOffset/dumpBitOffset/dumpBitWidth (added 2026-07-14): populates these
# dials from a Current Object Dump on connect/sync, not just from a live
# knob turn — owner: "we're not yet pulling the initial values from the
# patch sysex." Turned out Parameter Change writes (above) genuinely DO NOT
# show up in the same Current Object Dump reply (empirically confirmed:
# set Filter A to 5 via SysEx, re-requested the dump, byte-for-byte
# IDENTICAL to before — Parameter Change and Object Dump read from the same
# edit buffer per KRONOS_MIDI_SysEx.txt's own func 74 description, so this
# was surprising and worth double-checking rather than assuming a stale
# doc), so this was reverse-engineered via a bit-for-bit hardware diff
# instead of trusted from the doc alone.
#
# The reference doc's own Prog_EXi.txt table gives OFS/bit-range values
# RELATIVE to the start of each EXi slot's algorithm-specific block, not
# absolute offsets into the whole Program dump — e.g. "[Filter A] Cutoff"
# is documented at relative OFS 426, bits 7~5 continuing into 427 bits
# 3~0 (a 7-bit field, 0..99). The absolute base offset for each slot's
# algorithm-specific block was found in Prog_EXi_Common.txt (NOT
# Prog_EXi.txt) as the placeholder range immediately before the other
# slot's own "Common" section starts: EXi1's algorithm-specific block is
# OFS 2908-3908 (right before "EXi2 Common" at 3909), EXi2's is 3960-4959
# (right after "EXi2 Common" ends, matching the file's own "EXi Program
# Size: 4960 byte" header) — so absolute = 2908+relativeOFS for EXi1,
# 3960+relativeOFS for EXi2. Filter A Cutoff (rel 426) -> abs 3334 (EXi1) /
# 4386 (EXi2); Filter B Cutoff (rel 466) -> abs 3374 (EXi1) / 4426 (EXi2).
#
# The bit-packing itself is a continuous LSB-first bitstream spanning byte
# boundaries — same shape as Voyager's own dumpBitOffset/dumpBitWidth
# fields, just 8 usable bits per (already-decoded) byte instead of Moog's
# 7 — see read_korg_bitpacked_field()'s own comment, synthComms.c. All
# four offsets above were individually confirmed against real Kronos
# hardware (set 0 and a second distinct value via Parameter Change, diffed
# two Current Object Dump captures with tools/kronos_dump_diff.py, decoded
# with the same formula, got back the exact values sent) — not assumed
# from the doc's relative-offset arithmetic alone, given the doc's own
# "edit buffer" description turned out to need the empirical double-check
# above once already.
page       filters
section    al1
dialSize   50
spacing    60

color al1 0.85 0.4 0.3

dial al1FilterCutoff   label="AL-1 Filter A Cutoff (EXi1)" color=al1 max=100 typ=11 soc=20 sub=0 pid=4  idx=0 dumpOffset=3334 dumpBitOffset=5 dumpBitWidth=7
dial al1FilterCutoff2  label="AL-1 Filter A Cutoff (EXi2)" color=al1 max=100 typ=12 soc=20 sub=0 pid=4  idx=0 dumpOffset=4386 dumpBitOffset=5 dumpBitWidth=7
dial al1FilterBCutoff  label="AL-1 Filter B Cutoff (EXi1)" color=al1 max=100 typ=11 soc=20 sub=0 pid=49 idx=0 dumpOffset=3374 dumpBitOffset=4 dumpBitWidth=7
dial al1FilterBCutoff2 label="AL-1 Filter B Cutoff (EXi2)" color=al1 max=100 typ=12 soc=20 sub=0 pid=49 idx=0 dumpOffset=4426 dumpBitOffset=4 dumpBitWidth=7
